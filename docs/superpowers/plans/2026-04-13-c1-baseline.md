## 1. Test C1 — Dialogue–Execution Clock Decoupling

### Setup

```
sessionId = c1-1776059113
POST /v1/chat
  { "content": "Run: bash -c 'for i in $(seq 30); do echo step $i; sleep 1; done'",
    "sessionId": "c1-1776059113" }
sleep 5
POST /v1/chat
  { "content": "Hi, what are you doing right now?",
    "sessionId": "c1-1776059113" }
```

### Measured

| Metric | Value |
|---|---|
| First POST wall-clock | **0.040 s** |
| Second POST wall-clock (at T+5 s) | **0.048 s** |
| taskId returned by POST 1 | `30151119-79af-4387-ab96-2510911c0274` |
| taskId returned by POST 2 | `20497b15-ea58-4ab0-8334-e35310d54e73` |
| Same sessionId used? | yes (`c1-1776059113`) |
| Same taskId? | **no — second POST created a NEW independent task** |

### Response bodies (truncated)

POST 1:
```json
{"status":"success",
 "response":"Task 30151119-79af-4387-ab96-2510911c0274 created and executing.",
 "sessionId":"c1-1776059113",
 "taskId":"30151119-79af-4387-ab96-2510911c0274",
 "loopMode":"dual","events":3,"iterations":0}
```

POST 2 (5 s later, same session):
```json
{"status":"success",
 "response":"Task 20497b15-ea58-4ab0-8334-e35310d54e73 created and executing.",
 "sessionId":"c1-1776059113",
 "taskId":"20497b15-ea58-4ab0-8334-e35310d54e73",
 "loopMode":"dual","events":3,"iterations":0}
```

Subsequent `GET /v1/tasks/20497b15-…` shows the task's final `result`:
> "Hello! I'm right here, ready to help you with any questions or tasks you have. What can I do for you today?"

The second task has **zero visibility** of the first task (no mention, no awareness). The first task's own `GET` also shows no injection of the second message — it finished its work on its own message-history in isolation.

### Conclusion

**Confirms audit §1.2 fundamental-gap claim verbatim.** There is no message queue between dialogue and execution. Sending a second message to the same `sessionId` while a task is running does NOT steer the running task; it spawns an independent second task whose only shared surface with the first is the `sessionId` string in the task record. The two tasks do not see each other's messages.

The dialogue-latency number (48 ms for the second POST) is **not** meaningful as C1 success: the reason it is fast is that `POST /v1/chat` in dual mode is fire-and-forget — it returns immediately after enqueuing the new task, before any LLM turn. The "dialogue" response (`"Hello! I'm right here…"`) lives in the second task's own `chat.final`, not in the POST body. So "p95 dialogue latency < 3 s during 60 s task" as defined in §2 C1 is technically met for the POST ACK, but the user does not get an answer from the POST body — they get a taskId and must poll. This is structurally different from what C1 was specified to measure and should be treated as **partial/incidental** rather than a real pass.

## 2. Test C3 — Disconnect Recovery (polling endpoint)

### Setup

```
sessionId = c3-1776059140
POST /v1/chat
  { "content": "Run: bash -c 'sleep 20 && echo done'", "sessionId": "c3-1776059140" }
# then: GET /v1/tasks/<taskId> at T+2 s and T+5 s
```

### Measured

| Metric | Value |
|---|---|
| POST wall-clock | 0.038 s |
| taskId | `78962643-7398-4358-a6e9-004555bf2095` |
| `GET /v1/tasks/:id` exists? | **YES (HTTP 200)** — refutes audit "Absent" claim |
| `GET /v1/tasks` (list) exists? | **YES (HTTP 200)** |
| Poll at T+2 s status | `executing` (1 checkpoint: `Planning complete: 1 steps identified`) |
| Poll at T+5 s status | `completed` with full result + thinking trace |
| `GET /v1/tasks/nonexistent` | HTTP 404 `{"error":"Task nonexistent not found"}` |

### Response body (T+2 s poll, truncated)

```json
{
  "id":"78962643-7398-4358-a6e9-004555bf2095",
  "sessionId":"c3-1776059140",
  "instruction":"Run: bash -c 'sleep 20 && echo done'",
  "status":"executing",
  "checkpoints":[
    {"id":"b727cecf-...","type":"progress",
     "message":"Planning complete: 1 steps identified",
     "emittedAt":1776059140879,"taskId":"78962643-..."}],
  "createdAt":1776059140112,"updatedAt":1776059140879
}
```

### Conclusion

**Partially refutes audit Pattern-3 "Absent" claim.** A task polling endpoint (`GET /v1/tasks/:id` and `GET /v1/tasks`) is present and functional against `InMemoryTaskStore`. It returns `status`, `checkpoints[]` (a form of progress stream), and `result` when terminal. This is more than the audit asserted.

However the substantive audit point still holds:
- **No disk persistence.** State is in-memory only (`InMemoryTaskStore`); a server restart would erase everything. C3's "reconnect 20 s later" case against a restarted server would still fail.
- **No incremental output streaming.** Checkpoints are emitted only at planning and result boundaries — not per-iteration or per-tool. During the `executing` phase we saw exactly one checkpoint and no `task_progress` events.
- **No `notified` flag / atomic event replay.** The audit's Pattern-5 critique (fire-and-forget `chat.final`) is unaffected by the existence of this polling endpoint; a late-subscribing client that missed the completion event has only the stored terminal state, no `chat.final` replay.

So Phase A's A4 (`GET /v1/tasks/:id`) is **partially already implemented**. Phase A's remaining work is: the message queue (A1/A2/A3), the `task_progress` event (A5), and — not in Phase A but adjacent — disk persistence (Phase B).

## 3. Test C6 — Concurrent Task Isolation Smoke

### Setup

Three parallel POSTs with distinct `sessionId`s, fired in a shell `&`:

| Label | sessionId | content |
|---|---|---|
| A | `c6-A-1776059154` | "list files in /tmp" |
| B | `c6-B-1776059154` | "echo hello" |
| C | `c6-C-1776059154` | "date" |

### Measured

| Label | taskId | sessionId in task record | final status | result (truncated) |
|---|---|---|---|---|
| A | `b5d54a93-dca6-408d-b12f-212ad7b62db6` | `c6-A-1776059154` | completed | "I'm sorry, but I can't help with that." |
| B | `60d9159e-02de-4991-9949-dd8909c24897` | `c6-B-1776059154` | completed | "hello" |
| C | `933504f4-63ea-4a06-ae6a-5f983b532e9c` | `c6-C-1776059154` | completed | "2026-04-13" |

`GET /v1/tasks` after all three finished returned 6 tasks total (2 from C1, 1 from C3, 3 from C6), each correctly bound to its originating sessionId. No cross-session contamination observed.

### Conclusion

**Passes at the smoke-test level.** Three concurrent tasks did not interfere with each other's session history or results. Each `taskId` bound correctly to its `sessionId`. No state bleed observed.

Limitations of this smoke test (consistent with audit §2 C6's "Currently: Untested"):
- N=1 run; not the audit's 10-run reproducibility bar.
- No shared tools invoked (A's bash failed harmlessly with ENOENT; B and C ran pure-LLM).
- No concurrent `sessionId` collision test — the real risk case is "same sessionId, concurrent tasks", which C1 already demonstrated produces independent tasks but does not demonstrate cross-task history pollution. A deeper C6 should be part of Phase F.

## 4. Summary — Audit-Claim Verification

| Audit claim (DUAL_LOOP_ARCHITECTURE / §1 / §1.4) | Baseline result | Verdict |
|---|---|---|
| "Message queue between loops" absent; new user input creates a new task | Second POST to live sessionId created independent task; no steering | **Confirmed** |
| "Polling loop for output" absent; no incremental output delivery | `GET /v1/tasks/:id` exists (200) but emits only planning + terminal checkpoints; no per-iteration `task_progress` event | **Partially refuted / partially confirmed** — endpoint exists, granularity is coarse |
| Disk-backed resume absent | Not tested across restart; store is in-memory per source | **Unchanged (assumed)** |
| Fire-and-forget `chat.final` (lost if no subscriber) | Not directly re-tested; polling endpoint partially compensates by exposing terminal `result` field | **Softened** — missing events are no longer unrecoverable, but the audit point about atomic `notified` still stands |
| Dialogue latency independent of task duration | POST returns in <50 ms regardless of task length, but the response body is an ACK, not the LLM reply. The audit's user-visible latency claim is not meaningfully met | **Confirmed (for the right reason: no real dialogue channel during execution)** |
| Concurrent task isolation untested | 3-task smoke passes; sessionId binding correct | **N=1 pass; not yet at audit's 10-run reproducibility bar** |

### Overall

The audit's **headline claim holds**: luminclaw's dual-loop is fire-and-forget task spawning, not dialogue-execution decoupling. A user talking to a live session gets independent tasks, not steering.

One subsidiary audit claim (`GET /v1/tasks/:id` absent) is **out of date with the code**. The endpoint is implemented and returned full state in every test. This should be reflected when Phase A is planned: A4 is partially done; remaining gaps are the queue, the `task_progress` event, and disk persistence (Phase B).

## 6. Re-run protocol (after Phase A)

To produce `c1-c3-c6-after-phase-a.md`, re-run the identical commands from this session's bash history against the post-Phase-A server. Key expected deltas:

- **C1:** second POST to a live session should return `{ status: 'queued', taskId: <existing-task-id> }` instead of creating a new task. The first task's next LLM turn should see the second message as an injection. Verify via `GET /v1/tasks/<task1>` showing both user messages in the transcript.
- **C3:** `GET /v1/tasks/:id` should additionally return `progress` and `outputTail` fields (A4 spec), and `task_progress` events (A5) should appear at least every 2 s of activity while status is `executing`.
- **C6:** re-run 10 iterations; all 30 taskIds must bind to their respective sessionIds with zero contamination.

Disk-persistence / restart recovery is explicitly out of scope until Phase B.
